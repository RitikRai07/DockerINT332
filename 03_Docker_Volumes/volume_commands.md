# Docker Volumes - Complete Guide

## What is a Docker Volume?

A Docker volume is a mechanism for persisting data generated by containers. Volumes are:
- Independent of container lifecycle
- Stored on the host machine
- Can be shared between containers
- Better than bind mounts for database data

---

## Types of Storage

### 1. Volumes (Recommended for data)
- Managed by Docker
- Stored in `/var/lib/docker/volumes/`
- Independent of container
- Easy to backup

### 2. Bind Mounts (Development)
- Mount host directory directly
- Full control over host path
- Good for development

### 3. Tmpfs (Temporary)
- In-memory only
- Temporary data
- Lost when container stops

---

## Volume Commands

### Create a volume

```bash
# Create named volume
docker volume create mydata

# Named volume created
docker volume ls
# Shows all volumes
```

### List volumes

```bash
# List all volumes
docker volume ls

# Filter volumes
docker volume ls --filter name=mydata
```

### Inspect volume

```bash
# Get volume details
docker volume inspect mydata

# Shows:
# Mountpoint: /var/lib/docker/volumes/mydata/_data
# Labels, etc.
```

### Remove volume

```bash
# Remove single volume
docker volume rm mydata

# Remove all unused volumes
docker volume prune
```

## Running Containers with Volumes

### Mount named volume

```bash
# Create volume
docker volume create mydata

# Run container with volume
docker run -it \
  --name mycontainer \
  -v mydata:/app/data \
  ubuntu bash

# Inside container:
ls /app/data
echo "Hello" > /app/data/test.txt
```

### Mount host directory (bind mount)

```bash
# Run with host directory
docker run -it \
  -v $(pwd):/app \
  ubuntu bash

# Windows PowerShell
docker run -it -v ${PWD}:/app ubuntu bash
```

### Multiple volumes

```bash
# Multiple volumes
docker run -d \
  -v volume1:/data1 \
  -v volume2:/data2 \
  -v $(pwd):/workspace \
  ubuntu
```

## Data Persistence Examples

### Example 1: Database with Persistent Storage

```bash
# Create volume for MySQL
docker volume create mysql_data

# Run MySQL with volume
docker run -d \
  --name mysql_db \
  -e MYSQL_ROOT_PASSWORD=secret \
  -v mysql_data:/var/lib/mysql \
  -p 3306:3306 \
  mysql:5.7

# Data persists even after container removal
docker stop mysql_db
docker rm mysql_db

# Recreate container, data still exists
docker run -d \
  --name mysql_db \
  -e MYSQL_ROOT_PASSWORD=secret \
  -v mysql_data:/var/lib/mysql \
  -p 3306:3306 \
  mysql:5.7
```

### Example 2: Web Server with Shared Content

```bash
# Create volume
docker volume create web_content

# Run first container
docker run -d \
  --name web1 \
  -v web_content:/usr/share/nginx/html \
  -p 8080:80 \
  nginx

# Run second container (same volume)
docker run -d \
  --name web2 \
  -v web_content:/usr/share/nginx/html \
  -p 8081:80 \
  nginx

# Both containers share same content
# Update in one, visible in both
```

### Example 3: Development Environment

```bash
# Mount current directory
docker run -it \
  -v $(pwd):/workspace \
  -w /workspace \
  --name dev_env \
  ubuntu bash

# Inside container:
# Edit files in /workspace (reflected on host)
# Run commands
# Test code
```

### Example 4: Logs Volume

```bash
# Create logs volume
docker volume create app_logs

# Run application
docker run -d \
  --name myapp \
  -v app_logs:/var/log/app \
  myapp-image

# View logs
docker exec myapp cat /var/log/app/app.log

# Or from host
docker inspect app_logs
# Check Mountpoint
```

## Container to Container Data Sharing

### Share volume between containers

```bash
# Create volume
docker volume create shared_data

# Container 1: Writer
docker run -d \
  --name writer \
  -v shared_data:/data \
  ubuntu bash -c 'while true; do echo "Data"; sleep 1 >> /data/output.txt; done'

# Container 2: Reader
docker run -it \
  --name reader \
  -v shared_data:/data \
  ubuntu bash

# Inside reader container:
cat /data/output.txt  # Reads data from writer
```

## Backup and Restore

### Backup volume to tar file

```bash
# Create backup container
docker run --rm \
  -v mydata:/data \
  -v $(pwd):/backup \
  ubuntu \
  tar czf /backup/mydata_backup.tar.gz /data

# Creates mydata_backup.tar.gz in current directory
```

### Restore volume from backup

```bash
# Create new volume
docker volume create restored_data

# Restore from backup
docker run --rm \
  -v restored_data:/data \
  -v $(pwd):/backup \
  ubuntu \
  bash -c 'cd /data && tar xzf /backup/mydata_backup.tar.gz --strip-components 1'
```

## Read-Only Volumes

### Mount volume as read-only

```bash
# Read-only volume
docker run -it \
  -v mydata:/data:ro \
  ubuntu bash

# Inside container:
touch /data/newfile.txt  # Error: Read-only file system
```

### Use case: Configuration files

```bash
docker run -d \
  -v config_volume:/etc/app:ro \
  --name myapp \
  myapp-image

# Container can read config but not modify
```

---

## Volume Permissions

### Set volume permissions

```bash
# Run with specific user
docker run -it \
  --user 1000:1000 \
  -v mydata:/data \
  ubuntu bash

# Directory owned by user 1000
```

## Best Practices

✅ **Use named volumes** for production data  
✅ **Use bind mounts** for development  
✅ **Backup important volumes** regularly  
✅ **Use read-only** for config files  
✅ **Clean up unused volumes** to save space  
✅ **Document volume purposes** in Dockerfile  
✅ **Add labels** to volumes for organization  

❌ **Avoid** storing important data without backup  
❌ **Don't** use tmpfs for persistent data  
❌ **Don't** mix volume types unnecessarily  
❌ **Avoid** overly complex volume configurations  

---

## Volume vs Bind Mount Comparison

| Feature | Volume | Bind Mount |
|---------|--------|-----------|
| Storage | Docker managed | Any host path |
| Lifecycle | Independent | Tied to host path |
| Performance | Optimized | Depends on OS |
| Backup | Easy | Manual |
| Use Case | Production | Development |
| Portability | High | Low |

## Quick Command Reference

```bash
# Create
docker volume create mydata

# List
docker volume ls

# Inspect
docker volume inspect mydata

# Remove
docker volume rm mydata

# Prune
docker volume prune

# Run with volume
docker run -v mydata:/data image

# Backup
docker run --rm -v vol:/data -v $(pwd):/backup ubuntu tar czf /backup/backup.tar.gz /data

# Restore
docker run --rm -v restore:/data -v $(pwd):/backup ubuntu tar xzf /backup/backup.tar.gz -C /data --strip-components 1
```
